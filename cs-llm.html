Engagement Context

Automating Plain-Language Translation of Quantitative Portfolio Reports

A financial services client produced monthly portfolio performance reports consumed primarily by board members and relationship managers with no interest in interpreting model outputs. A quant analyst was spending 6 to 8 hours per monthly cycle manually translating performance metrics into narrative prose.

Six to eight hours per month, every month, on a step that was mostly mechanical. The framing shifted analyst to analyst, which introduced its own consistency problems. We automated the translation using the GPT-4 API, keeping the human in the loop for review but removing the drafting work entirely.

System Design

Pipeline Architecture and Prompt Engineering Approach

The pipeline takes a structured Python dictionary of monthly metrics and returns a four-section document: performance summary, attribution analysis, risk commentary, and forward outlook. Each section is a separate API call. That structure allows the analyst to review and approve sections independently and keeps any single prompt well within context limits.

  • System prompt encodes firm-specific terminology, risk appetite language, and the set of claims that require explicit hedging
  • Temperature set to 0.2 for factual stability — tested at 0.0 and 0.4 with 0.2 chosen as the balance point
  • Chain-of-thought instruction: model is asked to derive narrative from the data provided, not from pattern-matched boilerplate
  • Explicit instruction to flag anomalies in the data rather than smooth over them in narrative

Prompt Design

Engineering for Factual Accuracy in a Regulated Context

In a regulated financial context, fluency is not the risk. Wrong numbers are the risk. Forward-looking claims without qualifications are the risk. Inconsistent terminology that departs from what compliance has approved is the risk. The prompt engineering work was almost entirely about failure mode control, not about making the output read well.

The most valuable single decision was instructing the model to flag data anomalies rather than incorporate them silently. In the first production run, it surfaced a data entry error in the attribution table. Without that instruction, the error would have gone straight into the narrative.
Key prompt constraints
  • All numerical claims must cite the specific input field they derive from (traceability requirement)
  • Forward-looking statements must be framed as conditional on stated assumptions, never as predictions
  • Regulatory trigger phrases ("guaranteed", "will", "expected to") explicitly forbidden
  • Any input metric outside two standard deviations of the prior 12-month range must be flagged for analyst review rather than incorporated silently
  • Firm-specific terminology glossary injected into system prompt; model instructed to use client terms only
Evaluation Methodology

Measuring Output Quality and Factual Fidelity

You can't evaluate a financial LLM pipeline by reading it and deciding it sounds good. A scoring rubric was developed with the client's compliance team covering numerical accuracy, regulatory language, terminology adherence, and completeness. Every section was scored across six test reports before the pipeline went to production.

Reg. violations

The human review step stayed. That was a contractual and regulatory requirement. But what used to take 6 to 8 hours now takes under 20 minutes. The analyst reads and approves rather than drafts.

  • Pipeline depends on data structure remaining consistent; schema changes require prompt updates and re-evaluation
  • Model behavior on novel market conditions has not been tested; human review is the control for this risk
  • Terminology adherence at 94% reflects occasional near-synonyms not in the glossary; active monitoring recommended
  • GPT-4 API cost scales linearly with report length; immaterial at current volume but should be reviewed if volume increases
