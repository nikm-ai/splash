Context

Consolidating Fragmented Data into Tested, Version-Controlled Layers

Three separate clients had the same problem in slightly different forms: data scattered across systems that weren't designed to talk to each other, transformation logic living in spreadsheet formulas or SQL scripts nobody had reviewed in months, and downstream dashboards quietly consuming outputs nobody had tested.

In each case, a dbt Core project brought the transformation work into version control, separated it into well-defined layers, and left the client with a test suite they could actually run. The documentation transferred at project close rather than evaporating with the consultant.

Source Integration

Salesforce, Postgres, and Google Sheets as Unified Sources

Each source had its own failure mode. Salesforce objects came in via Fivetran with non-deterministic sync timing and occasional duplicate rows. Postgres tables were clean structurally but needed extensive type coercion. Google Sheets — which operations teams used for territory mappings and targets — was the most fragile: a stale manual update could silently corrupt downstream reports without a freshness check in place.

  • Salesforce: deduplication via row_number() over (partition by id order by _fivetran_synced desc) applied at every staging model
  • Postgres: cast applied uniformly at staging; no business logic until intermediate layer
  • Google Sheets: freshness block with 24-hour warn threshold; staleness breaks the DAG before downstream models run
  • All source declarations in a single sources.yml with full column-level documentation
Testing Framework

Automated Testing Across All Mart Models

Every mart model shipped with a schema.yml test suite. Generic tests covered the structural basics — uniqueness, nulls, accepted values. Singular tests written in SQL caught the domain-specific cases that generic tests can't express: revenue figures that should never be negative, or churn dates that can't precede enrollment.

Test coverage was a contractual deliverable in two of the three engagements. Clients kept the test suite on handoff as a regression harness for future changes.

Outcomes & Handoff

Results and Client Knowledge Transfer

The mart models were built with Looker naming conventions in mind from the start. That reduced the LookML work considerably and meant the handoff document was the dbt project itself — not a separate piece of documentation that would immediately fall out of date.

  • All three clients stopped running ad-hoc SQL scripts outside version control
  • Consistent naming across mart models shortened the LookML work on the Looker side
  • Quarterly dbt test runs integrated into CI pipelines in two engagements
  • Project documentation auto-generated via dbt docs generate served as the handoff artifact
The dbt docs generate site served as the handoff document. Lineage graphs, column-level descriptions, and test results in one place — the internal team could update them without re-engagement.
