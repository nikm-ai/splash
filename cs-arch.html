Evaluation Objective

Selecting an LLM Architecture for Internal Document Classification

A client needed to classify a large internal document corpus against a taxonomy of roughly 40 categories. The question was which architecture to build on: LoRA fine-tuning of a 7B parameter base model, prompt-tuning via soft prefix injection, or embedding-based retrieval against a labeled reference index.

We evaluated all three against four criteria: accuracy on a held-out test set, estimated infrastructure cost over 12 months, operational complexity in production, and the ability to add new categories without retraining. That last criterion turned out to matter more than expected.

Recommended: embedding retrieval

Retrieval won on accuracy and came in at a fraction of the infrastructure cost. That inverts the usual expectation — fine-tuning is supposed to beat retrieval on narrow tasks. The reason it didn't here is worth understanding.

These 40 categories are not domain-exotic. They're well-represented in the base model's pretraining data, so the embedding space already separates them without any additional training. Fine-tuning yields the biggest gains where the base model genuinely lacks exposure to the domain. That wasn't the case here.
Retrieval Architecture Details

Embedding Index Design and Classification Logic

The reference index was built from a labeled set of 15–25 examples per category, embedded using text-embedding-3-small and stored in a Postgres table with the pgvector extension. At inference time, the query document is embedded and the top-5 nearest neighbors are retrieved by cosine similarity. Category assignment follows a weighted majority vote.

  • ~800 labeled examples across 40 categories (mean: 20 per category)
  • Rare categories (<10 examples) augmented with lightly paraphrased variants; flagged and excluded from accuracy evaluation
  • Confidence threshold of 0.65 required for automatic classification; below-threshold docs routed to human review
  • New categories added by appending labeled examples to the index; no retraining or redeployment required
The ability to add new categories by appending examples to the index was the deciding factor in the recommendation. The document taxonomy shifts as regulatory requirements change. With LoRA, every taxonomy update triggers a retraining cycle. With retrieval, it's a few labeled examples added to a Postgres table.

Recommendation Rationale

Decision Framework and Architectural Trade-offs

Embedding retrieval: recommended

Retrieval dominates on all four evaluation criteria for this task: accuracy, cost, operational simplicity, and adaptability. The recommendation followed directly.

This result is task-specific. LoRA is the right choice when the base model genuinely lacks domain exposure — clinical note summarization, specialized translation, domain-specific code generation. The lesson here is diagnostic, not general: check whether your embedding space already separates the target categories before committing to fine-tuning infrastructure.
LoRA: not recommended for this task

LoRA came in second on accuracy and first on cost and operational burden. The GPU training infrastructure and serving endpoint are meaningful ongoing commitments, and they didn't produce a meaningful accuracy improvement on this taxonomy.

Prompt-tuning: not recommended

Prompt-tuning was last on accuracy and barely improved over zero-shot. Whether that reflects a taxonomy too broad for soft prompt capacity or a labeled set too small to learn a useful prefix, the result was the same either way.
